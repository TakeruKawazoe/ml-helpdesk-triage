# ダミーデータ設計

## ファイル

- `tickets_dummy.csv`: 学習・モデル選定用
- `tickets_holdout.csv`: 学習側で使わない語句と文章による最終確認用

## 目的

業務問い合わせ・障害チケットの一次切り分けを支援する教師あり機械学習モデルの学習用データとして使う。

このデータは完全なダミーデータであり、実在する企業、インターン先、顧客、社員、問い合わせ内容とは関係しない。

## 列定義

| 列名 | 種類 | 説明 |
|---|---|---|
| `ticket_id` | ID | チケットを一意に識別するID |
| `inquiry_text` | 入力特徴量 | 問い合わせ本文 |
| `label_category` | 教師ラベル | 問い合わせカテゴリ |
| `label_priority` | 教師ラベル | 優先度 |
| `label_department` | 教師ラベル | 推奨担当部署 |
| `impact_scope` | 入力特徴量 | 影響範囲 |
| `requester_role` | 入力特徴量 | 問い合わせ者の役割 |
| `channel` | 入力特徴量 | 問い合わせ経路 |
| `template_group` | 分割単位 | 同じ意味を持つ言い換え文を同じグループとして扱うためのID |
| `data_origin` | 出所 | `synthetic_train`または`synthetic_holdout` |

## カテゴリ

| カテゴリ | 想定内容 |
|---|---|
| 勤怠 | 打刻、勤務表、休暇申請、勤怠集計 |
| 請求 | 請求書作成、再発行、税率、入金ステータス |
| 権限 | 権限付与、権限不足、閲覧範囲、承認者変更 |
| システム障害 | 画面エラー、バッチ失敗、登録失敗、性能低下 |
| ネットワーク | WiFi、VPN、拠点間通信、DNS、インターネット接続 |
| アカウント | ログイン、MFA、アカウント発行、ロック、削除 |
| データ連携 | CSV取込、API連携、夜間連携、マスタ反映 |
| 端末 | PC、プリンタ、カメラ、ブラウザ、端末紛失 |
| その他・対象外 | 空調、照明、備品など、既存のITカテゴリへ無理に分類しない問い合わせ |

## 優先度

| 優先度 | 判断基準 |
|---|---|
| High | 全社や複数部署に影響する、業務停止、締め処理に影響、セキュリティリスクがある |
| Middle | 個人または部署単位で業務に支障がある、調査や修正が必要 |
| Low | 操作方法の確認、軽微な表示修正、急ぎではない設定依頼 |

## 担当部署

| 担当部署 | 主な対象 |
|---|---|
| 総務 | 勤怠、勤務表、休暇申請 |
| 経理 | 請求書、入金、税率、取引先マスタ |
| 情シス | アカウント、権限、端末、社内ツール |
| 開発 | アプリケーション不具合、データ連携、API、バッチ |
| インフラ | ネットワーク、VPN、DNS、クラウド接続 |

`その他・対象外`は一次受付先として総務へ送る。これはIT担当部署への誤振り分けを避けるための初期ルールであり、実組織へ導入する際は受付担当部署に合わせて変更する。

## 現在のデータ件数

- 学習・モデル選定用: 540件
- 未知表現の最終確認用: 81件
- 9カテゴリ

## データ生成方法

`src/generate_dummy_data.py` で、完全なダミーデータを再生成できる。

```bash
python src/generate_dummy_data.py
```

現在の分布は以下の通り。

| ラベル | 件数 |
|---|---:|
| 学習用カテゴリ | 9カテゴリ x 60件 |
| 学習用High | 180件 |
| 学習用Middle | 180件 |
| 学習用Low | 180件 |
| 最終確認用カテゴリ | 9カテゴリ x 9件 |

学習用では、1つの意味テンプレートから2種類の言い換え文を作る。同じ`template_group`の文を学習側と検証側へ分けないことで、ほぼ同じ文章を覚えただけの見かけ上の高スコアを防ぐ。

`tickets_holdout.csv`には学習用と異なる問い合わせ全文と文型を収録する。業務用語そのものをすべて未知にすると、語彙ベースのTF-IDFでは意味を判断できないため、対象語の言い換えは学習側にも含める。このファイルはモデル選定には使わず、採用モデルを決めた後の未知表現評価にだけ使う。

## 学習タスク

### タスク1：カテゴリ分類

入力：

- `inquiry_text`
- `impact_scope`
- `requester_role`
- `channel`

出力：

- `label_category`

### タスク2：優先度分類

入力：

- `inquiry_text`
- `impact_scope`
- `requester_role`
- `channel`

出力：

- `label_priority`

Highの見逃しが業務上のリスクになるため、AccuracyだけでなくHighクラスのRecallを重視する。

### タスク3：担当部署分類

入力：

- `inquiry_text`
- `impact_scope`
- `requester_role`
- `channel`

出力：

- `label_department`

## 注意点

- 実務データではないため、モデル精度は実サービス品質を保証しない。
- 初期段階では、成果物としての設計力・実装力・評価方法を示すことを目的にする。
- 今後、公開データセットや追加ダミーデータを取り込めるように前処理を分離する。
- 誤分類を前提に、UI側で予測結果の修正とフィードバック保存ができる設計にする。
- 合成データでの高スコアは実務精度を示さない。実データが得られた段階で、固定評価データを置き換えて再計測する。
