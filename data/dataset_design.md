# ダミーデータ設計

## ファイル

- `tickets_dummy.csv`

## 目的

業務問い合わせ・障害チケットの一次切り分けを支援する教師あり機械学習モデルの学習用データとして使う。

このデータは完全なダミーデータであり、実在する企業、インターン先、顧客、社員、問い合わせ内容とは関係しない。

## 列定義

| 列名 | 種類 | 説明 |
|---|---|---|
| `ticket_id` | ID | チケットを一意に識別するID |
| `inquiry_text` | 入力特徴量 | 問い合わせ本文 |
| `label_category` | 教師ラベル | 問い合わせカテゴリ |
| `label_priority` | 教師ラベル | 優先度 |
| `label_department` | 教師ラベル | 推奨担当部署 |
| `impact_scope` | 入力特徴量 | 影響範囲 |
| `requester_role` | 入力特徴量 | 問い合わせ者の役割 |
| `channel` | 入力特徴量 | 問い合わせ経路 |

## カテゴリ

| カテゴリ | 想定内容 |
|---|---|
| 勤怠 | 打刻、勤務表、休暇申請、勤怠集計 |
| 請求 | 請求書作成、再発行、税率、入金ステータス |
| 権限 | 権限付与、権限不足、閲覧範囲、承認者変更 |
| システム障害 | 画面エラー、バッチ失敗、登録失敗、性能低下 |
| ネットワーク | WiFi、VPN、拠点間通信、DNS、インターネット接続 |
| アカウント | ログイン、MFA、アカウント発行、ロック、削除 |
| データ連携 | CSV取込、API連携、夜間連携、マスタ反映 |
| 端末 | PC、プリンタ、カメラ、ブラウザ、端末紛失 |

## 優先度

| 優先度 | 判断基準 |
|---|---|
| High | 全社や複数部署に影響する、業務停止、締め処理に影響、セキュリティリスクがある |
| Middle | 個人または部署単位で業務に支障がある、調査や修正が必要 |
| Low | 操作方法の確認、軽微な表示修正、急ぎではない設定依頼 |

## 担当部署

| 担当部署 | 主な対象 |
|---|---|
| 総務 | 勤怠、勤務表、休暇申請 |
| 経理 | 請求書、入金、税率、取引先マスタ |
| 情シス | アカウント、権限、端末、社内ツール |
| 開発 | アプリケーション不具合、データ連携、API、バッチ |
| インフラ | ネットワーク、VPN、DNS、クラウド接続 |

## 初期データ件数

- 96件
- 8カテゴリ x 12件

## 学習タスク

### タスク1：カテゴリ分類

入力：

- `inquiry_text`
- `impact_scope`
- `requester_role`
- `channel`

出力：

- `label_category`

### タスク2：優先度分類

入力：

- `inquiry_text`
- `label_category`
- `impact_scope`
- `requester_role`
- `channel`

出力：

- `label_priority`

Highの見逃しが業務上のリスクになるため、AccuracyだけでなくHighクラスのRecallを重視する。

### タスク3：担当部署分類

入力：

- `inquiry_text`
- `label_category`
- `label_priority`
- `impact_scope`
- `requester_role`
- `channel`

出力：

- `label_department`

## 注意点

- 実務データではないため、モデル精度は実サービス品質を保証しない。
- 初期段階では、成果物としての設計力・実装力・評価方法を示すことを目的にする。
- 今後、公開データセットや追加ダミーデータを取り込めるように前処理を分離する。
- 誤分類を前提に、UI側で予測結果の修正とフィードバック保存ができる設計にする。
